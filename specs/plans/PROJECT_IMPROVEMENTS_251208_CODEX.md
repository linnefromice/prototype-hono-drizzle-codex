## アーキテクチャ

- ルートごとにリポジトリとユースケースを直接 new しており、依存解決の仕組みが薄い。アプリ起動時に依存を束ねるコンテナ（DI/ファクトリ）を用意すると単体テストや差し替えが容易になる。
- 環境変数を各リクエストごとに loadEnvConfig で読み込んでおり、I/Oが毎回発生する。起動時に1度だけ読み込んで共有する構造にすると効率と一貫性が上がる。
- エラーハンドリングやレスポンスの統一レイヤーがなく、各ルートで個別対応している。共通ミドルウェアでロギング・エラー整形を行うと運用性が向上する（例: バリデーションエラー→400、予期せぬエラー→500 を共通化）。

## テストの仕組み

- items の統合テストはテスト開始時にテーブルを動的生成・TRUNCATEしている。テスト専用のマイグレーションやシードを用意し、テスト用 DB（DATABASE_URL 切替）を使うワークフローを整備すると再現性が高まる。
- 環境変数をテスト内で直接書き換えており、他のテストへの影響が懸念される。テスト用の設定ローダーやモックを導入すると安全性が上がる。
- OpenAPI との契約テストが無く、実装とスキーマの乖離を検知できない。生成された Zod スキーマでレスポンスを検証する E2E/統合テストを追加すると API 互換性を担保できる。

## ドキュメンテーション

- README はセットアップとコマンド手順が中心で、サービスの責務分割や依存関係、デプロイ方法などのアーキテクチャ概要が不足している。ディレクトリ構成、データフロー、依存図を追加すると新規参加者が理解しやすくなる。
- 環境変数の一覧・意味・既定値、テスト/本番での設定例が README に無い。.env のサンプルと併せて表形式でまとめると運用が楽になる。
- API の利用手順やバージョニング方針が明文化されていない。OpenAPI から生成したクライアントの使い方や変更時の合意プロセスをドキュメント化すると、将来のエンハンスに備えられる。
